1. 과제 개요

1.1. 개발 목적 및 필요성

본 기말과제의 주요 목적은 자바(Java) 언어의 핵심 개념인 객체지향 프로그래밍(OOP) 패러다임을 실제 애플리케이션 개발에 적용해 보는 것입니다. 단순히 콘솔 기반의 프로그램을 넘어, JavaFX 라이브러리를 활용하여 사용자와 상호작용할 수 있는 그래픽 사용자 인터페이스(GUI)를 구축하였습니다.

기존 수업 시간에 실습했던 단순한 사각형 및 원 그리기 예제를 발전시켜, 상용 그래픽 편집 도구에서 제공하는 필수 기능(직선, 자유곡선 그리기, 색상 및 굵기 변경, 실행 취소/다시 실행, 프로젝트 파일 저장/불러오기)을 모두 포함하는 확장형 그림판 애플리케이션을 완성하는 것을 목표로 삼았습니다. 이를 통해 자바의 컬렉션 프레임워크(ArrayList, Stack), 이벤트 드리븐 프로그래밍, 그리고 파일 입출력 메커니즘을 종합적으로 이해하고 실무적인 응용 능력을 배양하고자 합니다.

1.2. 개발 환경 및 기술 스택

본 애플리케이션은 다음과 같은 환경에서 개발되었습니다.

  • 운영체제 (OS): Windows 10/11
  • 프로그래밍 언어: Java (JDK 21)
  • GUI 프레임워크: JavaFX (이벤트 처리 및 캔버스 렌더링)
  • 통합 개발 환경 (IDE): Eclipse IDE
  • 형상 관리 (VCS): Git (버전 관리 및 코드 이력 추적)

2. 시스템 설계 및 아키텍처

2.1. 전체 패키지 구조

프로그램의 응집도를 높이고 모듈화를 달성하기 위해 코드를 두 개의 주요 패키지로 분리하여 설계하였습니다.

  • application 패키지: 프로그램의 흐름을 제어하고 UI를 구성하는 컨트롤러 및 매니저 클래스들이 위치합니다. (Main, ShapeManager, ToolManager, MyMouseHandler 등)
  • shape 패키지: 화면에 그려지는 시각적 객체들의 모델(Model)을 정의합니다. 객체지향의 상속 구조를 철저히 따르고 있습니다. (Shape, Rectangle, Circle, Line, FreeLine)

2.2. 주요 클래스 다이어그램 및 역할

핵심 비즈니스 로직을 담당하는 클래스의 역할은 다음과 같습니다.

  • Main.java: JavaFX의 Application 클래스를 상속받아 프로그램의 메인 윈도우(Stage)와 Scene을 생성합니다. BorderPane 레이아웃을 사용하여 상단에는 메뉴 및 툴바를, 중앙에는 Canvas를 배치합니다.
  • ShapeManager.java: 캔버스에 생성된 모든 Shape 객체들을 ArrayList 자료구조로 관리합니다. 화면을 갱신(`drawAll`)하거나 특정 좌표에 있는 도형을 찾는(`findShape`) 역할을 수행하며, Undo/Redo 기능을 위한 상태 저장소(`Stack`)를 관리합니다.
  • ToolManager.java: 현재 사용자가 선택한 그리기 모드(사각형, 원, 이동 등)와 선택된 색상, 선 굵기 등의 상태 데이터를 저장하는 컨텍스트(Context) 클래스입니다.

아래는 핵심이 되는 상태 관리 클래스의 일부분입니다.

public class ToolManager { public static final int DRAW_RECT = 1; public static final int DRAW_CIRCLE = 2; public static final int MOVE_SHAPE = 3; ... private int currentMode; private Color fillColor = Color.WHITE; private Color lineColor = Color.BLACK; private double lineWidth = 1.0; ... }

3. 주요 기능 및 구현 상세 (핵심 기술)

3.1. JavaFX 기반의 현대적 GUI 구성

사용자 편의성을 극대화하기 위해 상용 소프트웨어와 유사한 상단 `MenuBar`와 `ToolBar` 구조를 채택하였습니다. ColorPicker 컴포넌트를 사용하여 사용자가 자유롭게 RGB 기반의 색상을 선택할 수 있도록 하였으며, Slider 컴포넌트를 통해 1부터 10까지의 선 굵기를 시각적으로 조절할 수 있습니다.

// UI 구성 코드 일부 (Main.java) ToolBar toolBar = new ToolBar(); ColorPicker fillColorPicker = new ColorPicker(Color.WHITE); ColorPicker lineColorPicker = new ColorPicker(Color.BLACK); Slider lineWidthSlider = new Slider(1, 10, 1); toolBar.getItems().addAll( btnRect, btnCircle, btnLine, btnFreeLine, new Separator(), btnMove, btnClear, new Separator(), new Label("채우기:"), fillColorPicker, new Label("선:"), lineColorPicker, new Label("굵기:"), lineWidthSlider );

3.2. 객체지향 패러다임(상속과 다형성)을 활용한 도형 렌더링

도형 렌더링 시스템은 완벽한 다형성(Polymorphism)을 기반으로 동작합니다. 최상위 추상 클래스인 Shape는 모든 도형이 가져야 할 공통 속성(색상, 좌표, 굵기)과 추상 메서드(draw, move, isMyRange)를 정의합니다.

이러한 설계 덕분에 ShapeManager는 배열에 담긴 객체가 사각형인지 자유 곡선인지 알 필요 없이, 반복문을 통해 shape.draw(gc)만 호출하면 자바의 동적 바인딩(Dynamic Binding) 메커니즘에 의해 올바른 형태로 렌더링됩니다. 이는 개방-폐쇄 원칙(OCP)을 준수하는 훌륭한 설계입니다.

3.3. Stack 자료구조를 활용한 Undo / Redo 알고리즘

본 과제에서 가장 기술적으로 난이도가 높았던 부분은 실행 취소(Undo) 기능의 구현입니다. 사용자의 작업 내역을 추적하기 위해 자바의 java.util.Stack 자료구조를 활용했습니다.

단순히 명령어(Command)를 저장하는 방식 대신, 화면에 변화가 생길 때마다 도형 리스트(`ArrayList`) 전체를 직렬화(Serialize)하여 바이트 배열(`byte[]`) 형태로 복사(Deep Copy)한 뒤 스택에 Push하는 방식을 채택했습니다. 이 방식은 메모리를 약간 더 사용하지만, 그리기 상태 복원의 무결성을 100% 보장합니다.

// 상태 저장 로직 (ShapeManager.java) private void saveState() { byte[] data = serializeShapes(shapeList); // 리스트를 바이트 배열로 변환 if (data != null) { undoStack.push(data); // 이전 상태를 스택에 저장 redoStack.clear(); // 새로운 작업이 발생하면 Redo 스택은 초기화 } } public void undo() { if (undoStack.size() > 1) { redoStack.push(undoStack.pop()); // 현재 상태를 Redo로 이동 byte[] previousData = undoStack.peek(); shapeList = deserializeShapes(previousData); // 이전 상태 복원 } }

3.4. 객체 직렬화(Serialization)를 통한 파일 입출력

그림판에서 작업한 결과물을 디스크에 저장하고 불러오는 파일 입출력(File I/O) 기능을 구현하였습니다. 이를 위해 ObjectOutputStream과 ObjectInputStream을 사용하였습니다.

초기 코드 설계 당시 Shape 객체 내부에 JavaFX의 UI 컴포넌트인 GraphicsContext가 멤버 변수로 존재하여 NotSerializableException 예외가 발생하는 치명적인 문제가 있었습니다. 이를 해결하기 위해 GraphicsContext를 객체의 속성(Field)에서 분리하고, 그리기 행위가 일어나는 시점(`draw()` 메서 호출 시)에만 파라미터로 전달받도록 코드를 리팩토링 하였습니다. 이 과정을 통해 객체의 역할과 책임(SRP 원칙)을 더욱 명확히 할 수 있었습니다.

// 파일 저장 기능 로직 (Main.java 및 ShapeManager.java 연동) saveItem.setOnAction(e -> { File file = fileChooser.showSaveDialog(primaryStage); if (file != null) { shapeManager.saveToFile(file); // ObjectOutputStream을 활용한 저장 } }); // Shape.java 구조 개선 public abstract class Shape implements Serializable { private static final long serialVersionUID = 1L; // 직렬화를 위해 Color 객체 대신 R, G, B 원시값 사용 protected double fillR, fillG, fillB, fillA; ... // 기존의 멤버 변수 GraphicsContext gc; 제거됨. public abstract void draw(GraphicsContext gc); // 호출 시 전달받음 }

5. 결론 및 느낀 점

본 기말과제를 통해 캔버스 렌더링과 이벤트 처리라는 기본적인 그래픽 애플리케이션의 동작 원리를 명확히 이해할 수 있었습니다. 특히 초기 코드 베이스에서 상용 수준의 프로그램으로 발전시키는 과정에서, 단순한 기능 추가를 넘어 "어떻게 하면 기존 코드를 망가뜨리지 않고 확장(Scale-up)할 수 있을까?"라는 고민을 끊임없이 하였습니다.

가장 기억에 남는 어려움은 Undo/Redo 기능과 저장/불러오기 기능을 구현하기 위해 객체를 직렬화(Serialize)하는 과정에서 겪은 에러였습니다. JavaFX의 GraphicsContext와 Color 객체가 직렬화를 지원하지 않아 구조적인 한계에 부딪혔고, 이를 우회하기 위해 상태 데이터(RGB 실수 값)와 랜더링 도구(GraphicsContext)를 완벽히 분리해내는 리팩토링 과정을 거쳤습니다. 이 과정을 통해 책에서만 배우던 객체지향의 단일 책임 원칙(SRP)과 데이터와 로직의 분리의 중요성을 피부로 느낄 수 있었습니다.

또한, Stack 자료구조를 단순히 알고리즘 문제를 푸는 데 그치지 않고, 사용자의 행위(History)를 추적하고 복원하는 실무적인 아키텍처 패턴으로 응용해 볼 수 있어 매우 뜻깊은 과제였습니다. 이번 프로젝트를 성공적으로 완수하면서 자바 프로그래밍에 대한 자신감이 크게 향상되었으며, 추후 디자인 패턴(Design Pattern)을 더 깊게 공부하여 현재 코드에 커맨드(Command) 패턴을 완벽히 도입해 보고 싶다는 새로운 목표도 생겼습니다.
